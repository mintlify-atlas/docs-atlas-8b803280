MCRIT occasionally requires database migrations when updates to core algorithms (MinHash, PicHash) or data structures are released. This guide covers migration procedures for major versions.

## Overview

MCRIT stores calculated hashes (MinHash and PicHash) in the database. When the underlying algorithms change, these hashes must be recalculated to maintain compatibility and accuracy.

### When Migration is Required

Migration is necessary when:

- **MinHash algorithm changes**: Affects function similarity matching
- **PicHash algorithm changes**: Affects position-independent code matching  
- **SMDA version updates**: Changes in disassembly/instruction escaping
- **Band indexing changes**: LSH index structure modifications

### Version Compatibility

MCRIT tracks compatibility through:
- **SMDA version** in `SampleEntry.smda_version`
- **Configuration hashes** from `MinHashConfig.getConfigHash()`
- **Database state** tracked in storage metadata

## Migration for v1.3.0

With MCRIT v1.3.0, several SMDA improvements were released that affect hash calculation:

- Fixed address masking issues ([SMDA #37](https://github.com/danielplohmann/smda/issues/37))
- Fixed unhandled instruction escaping ([SMDA #48](https://github.com/danielplohmann/smda/issues/48))
- Fixed opcode byte parsing ([SMDA #46](https://github.com/danielplohmann/smda/issues/46))
- Fixed Delphi binary parsing stalls ([SMDA #44](https://github.com/danielplohmann/smda/issues/44))

These fixes improve accuracy but change PicHash and MinHash values for affected functions.

### Impact

**PicHash changes**:
- Functions with improperly masked addresses will get new PicHashes
- Previously missed identical functions will now match

**MinHash changes**:
- Instruction escaping improvements affect shingling
- Similarity scores may change slightly

### Prerequisites

Before migrating, ensure you have the correct versions:

```bash
# Update requirements
pip install -r requirements.txt

# Verify SMDA version
pip freeze | grep smda
# Should show: smda==1.3.11 or higher
```

### Migration Procedure

The migration recalculates hashes **without reprocessing binaries**, using stored disassembly data.

#### Option 1: Using curl

Queue recalculation jobs via REST API:

```bash
# Recalculate all PicHashes
curl http://127.0.0.1:8000/recalculate_pichashes

# Recalculate all MinHashes  
curl http://127.0.0.1:8000/recalculate_minhashes
```

#### Option 2: Using McritClient

Programmatically queue jobs:

```python
from mcrit.client.McritClient import McritClient

# Connect to MCRIT server
client = McritClient(mcrit_server="http://127.0.0.1:8000")

# Queue PicHash recalculation
pichash_job = client.recalculatePicHashes()
print(f"PicHash recalculation job: {pichash_job['job_id']}")

# Queue MinHash recalculation
minhash_job = client.recalculateMinHashes()
print(f"MinHash recalculation job: {minhash_job['job_id']}")
```

#### Option 3: Using McritWeb (Admin)

For deployments with MCRITweb frontend:

1. Log in as admin user
2. Navigate to **Server** section
3. Click **Recalculate PicHashes**
4. Click **Recalculate MinHashes**
5. Monitor job progress in **Jobs** section

### Monitoring Progress

Recalculation runs as asynchronous jobs. Monitor via:

#### CLI

```bash
# Check job status
mcrit client job-status <job_id>

# List all jobs
mcrit client jobs
```

#### Python

```python
import time
from mcrit.client.McritClient import McritClient

client = McritClient()

# Poll job status
job_id = pichash_job['job_id']
while True:
    status = client.getJobStatus(job_id)
    print(f"Job {job_id}: {status['state']} - {status.get('progress', 0)}%")
    
    if status['state'] in ['finished', 'failed', 'aborted']:
        break
    
    time.sleep(10)
```

#### REST API

```bash
# Get job status
curl http://127.0.0.1:8000/jobs/<job_id>

# List all jobs
curl http://127.0.0.1:8000/jobs
```

### Performance Expectations

Recalculation time depends on database size:

- **Small DB** (< 1,000 functions): Minutes
- **Medium DB** (1,000 - 100,000 functions): Hours  
- **Large DB** (> 100,000 functions): Hours to days

Progress is reported as percentage complete.

### Post-Migration Validation

```python
from mcrit.client.McritClient import McritClient

client = McritClient()

# Check system status
status = client.getStatus()
print(f"Samples: {status['num_samples']}")
print(f"Functions: {status['num_functions']}")
print(f"PicHashes: {status['num_pichashes']}")

# Verify a sample's SMDA version was updated
sample = client.getSampleById(sample_id=1)
print(f"SMDA version: {sample['smda_version']}")
# Should show updated version (e.g., 1.3.11)
```

## Migration for v1.4.6 (Linear Banding)

Version 1.4.6 introduced linear banding strategy as the new default, replacing random banding.

### Changes

**Linear banding** provides:
- More predictable candidate selection
- Better distribution across signature space
- Deterministic band assignments

### Migration Required

**Complete index rebuild** is required when switching banding strategies:

```python
from mcrit.client.McritClient import McritClient

client = McritClient()

# Rebuild MinHash band index with new strategy
rebuild_job = client.rebuildMinHashIndex()
print(f"Rebuild job: {rebuild_job['job_id']}")
```

### Configuration Changes

Update `StorageConfig` to use linear banding:

```python
# config/StorageConfig.py
STORAGE_BAND_STRATEGY = "linear"  # Changed from "random"
STORAGE_BANDS = {4: 16}  # Must satisfy: band_size * num_bands = MINHASH_SIGNATURE_LENGTH
```

**Important**: With linear strategy:
- Use a single band size
- Ensure `band_size × num_bands = MINHASH_SIGNATURE_LENGTH`
- Example: 4 × 16 = 64 (default signature length)

## Migration Best Practices

### 1. Backup Before Migration

**MongoDB backup**:
```bash
# Dump entire database
mongodump --db=mcrit --out=/backup/mcrit_pre_migration

# Restore if needed
mongorestore --db=mcrit /backup/mcrit_pre_migration/mcrit
```

**Export MCRIT data**:
```python
from mcrit.client.McritClient import McritClient

client = McritClient()

# Export all samples
export_data = client.getExportData()

# Save to file
import json
with open('mcrit_export.json', 'w') as f:
    json.dump(export_data, f)
```

### 2. Test on Subset

For large databases, test migration on a subset:

```python
# Create test instance with sample data
test_client = McritClient(mcrit_server="http://test-server:8000")

# Import small subset
test_client.importData(sample_subset)

# Run migration
test_client.recalculatePicHashes()
test_client.recalculateMinHashes()

# Validate results
```

### 3. Schedule Downtime

For production systems:
- Announce maintenance window
- Stop accepting new submissions during migration
- Keep workers running to process recalculation jobs

### 4. Monitor Resources

```bash
# Monitor worker CPU/memory
top -p $(pgrep -f "mcrit worker")

# Monitor MongoDB
mongotop --host 127.0.0.1:27017

# Monitor disk space
df -h
```

## Troubleshooting

### Migration Job Stuck

**Check worker status**:
```bash
# Ensure workers are running
ps aux | grep "mcrit worker"

# Check worker logs
tail -f /var/log/mcrit/worker.log
```

**Restart workers** if hung:
```bash
# Stop workers
pkill -f "mcrit worker"

# Start new workers
mcrit worker &
```

### Out of Memory

**Reduce batch sizes** in configuration:
```python
# MinHashConfig
MINHASH_GENERATION_WORKPACK_SIZE = 5000  # Reduce from 10000
MINHASH_BAND_REBUILD_WORK_PACKAGE_SIZE = 50000  # Reduce from 100000
```

**Use SpawningWorker** to isolate memory:
```bash
# Use spawning worker instead of standard worker  
mcrit spawningworker
```

### Inconsistent Results

**Clear query samples** that may be outdated:
```python
client = McritClient()
client.clearQuerySamples()
```

**Verify SMDA versions** match:
```python
samples = client.getSamples()
versions = set(s['smda_version'] for s in samples)
print(f"SMDA versions in database: {versions}")
# Should all be updated to latest version
```

## Version-Specific Notes

### v1.2.26 → v1.3.0
- **Required**: PicHash and MinHash recalculation
- **Reason**: SMDA 1.3.11 fixes
- **Estimate**: ~1 hour per 10,000 functions

### v1.3.x → v1.4.6  
- **Required**: Band index rebuild (if using linear banding)
- **Optional**: MinHash recalculation (no algorithm changes)
- **Estimate**: ~30 minutes per 100,000 functions

### v1.4.5 → v1.4.6
- **Optional**: Switch to linear banding (recommended)
- **Required if switching**: Complete band index rebuild
- **Estimate**: ~20 minutes per 100,000 functions

## Automated Migration Scripts

For recurring migrations, create automation:

```python
#!/usr/bin/env python3
"""MCRIT Migration Script"""
import time
import sys
from mcrit.client.McritClient import McritClient

def migrate_to_v1_3_0():
    client = McritClient()
    
    print("Starting v1.3.0 migration...")
    
    # Step 1: Backup
    print("Step 1: Creating backup...")
    # Implement backup logic
    
    # Step 2: Recalculate PicHashes
    print("Step 2: Recalculating PicHashes...")
    pichash_job = client.recalculatePicHashes()
    wait_for_job(client, pichash_job['job_id'])
    
    # Step 3: Recalculate MinHashes
    print("Step 3: Recalculating MinHashes...")
    minhash_job = client.recalculateMinHashes()
    wait_for_job(client, minhash_job['job_id'])
    
    # Step 4: Validate
    print("Step 4: Validating migration...")
    status = client.getStatus()
    print(f"Functions: {status['num_functions']}")
    print(f"PicHashes: {status['num_pichashes']}")
    
    print("Migration complete!")

def wait_for_job(client, job_id, poll_interval=10):
    """Wait for job to complete"""
    while True:
        job = client.getJobStatus(job_id)
        state = job['state']
        progress = job.get('progress', 0)
        
        print(f"  Job {job_id}: {state} ({progress}%)")
        
        if state == 'finished':
            return True
        elif state in ['failed', 'aborted']:
            print(f"  Job failed: {job.get('error', 'Unknown error')}")
            return False
        
        time.sleep(poll_interval)

if __name__ == '__main__':
    migrate_to_v1_3_0()
```
