The recommended way to deploy MCRIT is using the fully packaged Docker solution, which provides a complete, production-ready environment with minimal configuration.

## Why Use Docker?

The Docker deployment offers several advantages:

- **Pre-configured Components**: All components are pre-configured and tested to work together
- **Version Compatibility**: Ensures fully compatible versions across MCRIT, MongoDB, and the web frontend
- **Easy Deployment**: Get started with just a few commands
- **Isolated Environment**: Runs in containers without affecting your system
- **Persistent Storage**: Includes MongoDB for data persistence
- **Web Interface**: Comes with a web frontend for convenient interaction

## Using docker-mcrit

<Steps>
  <Step title="Clone the docker-mcrit repository">
    The official Docker deployment is maintained in a separate repository:
    
    ```bash
    git clone https://github.com/danielplohmann/docker-mcrit.git
    cd docker-mcrit
    ```
  </Step>

  <Step title="Review the configuration">
    Check the `docker-compose.yml` file and any configuration files to ensure they match your requirements. The default configuration should work for most use cases.
  </Step>

  <Step title="Start the containers">
    Use Docker Compose to start all services:
    
    ```bash
    docker-compose up -d
    ```
    
    This will start:
    - MCRIT server (REST API)
    - MCRIT worker(s)
    - MongoDB database
    - Web frontend (if included)
  </Step>

  <Step title="Verify the deployment">
    Check that all containers are running:
    
    ```bash
    docker-compose ps
    ```
    
    Access the web interface (typically at `http://localhost:8000` or the configured port) to verify the deployment is working.
  </Step>
</Steps>

## Architecture

The Docker deployment typically includes:

### MCRIT Server

- Provides the REST API interface on port 8000 (default)
- Handles client requests and job scheduling
- Connects to MongoDB for data storage

### MCRIT Worker

- Processes queued jobs from the server
- Can be scaled horizontally by running multiple worker containers
- Performs CPU-intensive tasks like MinHash calculation and matching

### MongoDB

- Provides persistent storage for samples, functions, and analysis results
- Stores the job queue and execution state
- Default database name: `mcrit`

### Web Frontend

- User-friendly interface for interacting with MCRIT
- Visualizes analysis results and relationships
- Manages samples and families

## Common Operations

### Viewing Logs

```bash
# View logs for all services
docker-compose logs

# View logs for a specific service
docker-compose logs mcrit-server
docker-compose logs mcrit-worker
docker-compose logs mongodb

# Follow logs in real-time
docker-compose logs -f
```

### Stopping Services

```bash
# Stop all containers
docker-compose down

# Stop and remove volumes (WARNING: deletes all data)
docker-compose down -v
```

### Scaling Workers

```bash
# Run multiple worker instances for parallel processing
docker-compose up -d --scale mcrit-worker=4
```

### Accessing the MongoDB Shell

```bash
# Connect to MongoDB container
docker-compose exec mongodb mongosh mcrit
```

## Environment Variables

The Docker deployment can be customized using environment variables in a `.env` file or directly in `docker-compose.yml`:

- `MONGODB_PORT`: MongoDB port (default: 27017)
- `API_PORT`: MCRIT API port (default: 8000)
- `MONGODB_USERNAME`: MongoDB authentication username
- `MONGODB_PASSWORD`: MongoDB authentication password

## Data Persistence

By default, MongoDB data is stored in a Docker volume, ensuring persistence across container restarts. To back up your data:

```bash
# Create a MongoDB dump
docker-compose exec mongodb mongodump --db mcrit --out /backup

# Copy the backup from the container
docker cp <container-id>:/backup ./mcrit-backup
```

## Troubleshooting

### Container Won't Start

Check the logs for error messages:

```bash
docker-compose logs <service-name>
```

### Port Already in Use

If port 8000 or 27017 is already in use, modify the port mappings in `docker-compose.yml`.

### Worker Not Processing Jobs

Ensure the worker can connect to MongoDB and the job queue:

```bash
docker-compose logs mcrit-worker
```
