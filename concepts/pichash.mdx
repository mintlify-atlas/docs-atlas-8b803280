PicHash (Position-Independent Code Hash) is MCRIT's mechanism for exact matching of functions based on their position-independent code representation. Unlike MinHash which provides fuzzy similarity, PicHash identifies identical or near-identical code regardless of where it's loaded in memory.

## Overview

PicHash complements MinHash by handling scenarios where exact or near-exact matches are needed:

- **Library function detection**: Identifying statically linked library code
- **Code reuse**: Finding exact copies of functions across samples
- **Small functions**: Efficiently matching functions too small for MinHash
- **High-confidence matches**: Providing 100% match scores for identical code

## What is Position-Independent Code?

Position-independent code (PIC) is code that executes correctly regardless of its absolute address in memory. In reverse engineering, we achieve this by:

1. **Normalizing addresses**: Replace absolute addresses with relative offsets or generic placeholders
2. **Abstracting registers**: Use generic register representations
3. **Standardizing immediates**: Normalize immediate values where appropriate

This normalization is performed by SMDA (Smart Disassembly Annotation) before MCRIT processes the code.

## PicHash Generation

PicHash values are generated during SMDA disassembly and stored in function entries:

```python

        # ...

            self.pichash = smda_function.pic_hash
            self.picblockhashes = []
```

Source: `mcrit/storage/FunctionEntry.py:34-62`

The `pic_hash` is computed by SMDA during function analysis and represents a hash of the position-independent code representation.

<Info>
PicHash values are calculated by SMDA, not MCRIT. MCRIT indexes and queries these pre-computed hashes for matching.
</Info>

## PicHash vs MinHash

| Aspect | PicHash | MinHash |
|--------|---------|----------|
| **Match Type** | Exact | Fuzzy (similarity) |
| **Score** | 100% or no match | 0-100% similarity |
| **Collision Resistance** | High (cryptographic hash) | Intentional collisions |
| **Small Functions** | Excellent | Poor (needs ≥10 instructions) |
| **Variants** | Misses minor changes | Tolerates modifications |
| **Speed** | Very fast (hash lookup) | Fast (LSH candidates) |
| **Use Case** | Library detection, exact copies | Similar code, refactored versions |

### Match Harmonization

PicHash matches are processed into a standardized format:

```python
def _harmonizePicHashMatches(self, pichash_matches):
    pichash_match_mapping = {}
    sample_function_id_to_size = {
        entry.function_id: entry.num_instructions 
        for entry in self._function_entries
    }
    
    for _, pichash_tuples in pichash_matches.items():
        current_own_function_ids = []
        current_foreign_function_ids = []
        function_size = 0
        
        for pichash_tuple in pichash_tuples:
            bh_family_id, bh_sample_id, bh_function_id = pichash_tuple
            if bh_function_id in sample_function_id_to_size:
                current_own_function_ids.append(bh_function_id)
                function_size = sample_function_id_to_size[bh_function_id]
            else:
                current_foreign_function_ids.append(bh_function_id)
        
        # Only match if function meets size threshold
        if function_size < self._pichash_size:
            continue
            
        # Create match entries with 100% score
        if len(pichash_tuples) > 1:
            for own_function_id in current_own_function_ids:
                for foreign_function_id in current_foreign_function_ids:
                    foreign_sample_id = cached_function_id_to_sample_id[foreign_function_id]
                    pichash_match_mapping[(own_function_id, foreign_sample_id, foreign_function_id)] = 
                        (100, True, False)  # score=100, is_pichash=True, is_minhash=False
```

## Size Threshold

PicHash matching includes a configurable size threshold to avoid matching trivial functions:

```python
# In MinHashConfig
PICHASH_SIZE: int = 10  # minimum function size for PicHash matching
```

Functions with fewer than 10 instructions (by default) are excluded from PicHash matching, even if they share the same hash.

<Note>
The `PICHASH_SIZE` threshold prevents matching of very small functions like single-instruction stubs that might coincidentally have the same PicHash.
</Note>

## PicHash and MinHash Integration

MCRIT combines both hashing methods for comprehensive matching:

```python
# Configuration option
PICHASH_IMPLIES_MINHASH_MATCH: bool = True
```

When enabled (default), PicHash matches are treated as implied MinHash matches:

```python
if self._worker._minhash_config.PICHASH_IMPLIES_MINHASH_MATCH:
    for key, new_entry in pichash_matches.items():
        if key not in minhash_matches:
            minhash_matches[key] = (100.0, 0, 1)  # Treat as MinHash match
```

Source: `mcrit/matchers/MatcherInterface.py:564-567`

This optimization reduces computational overhead:
1. PicHash matches are found first (fast hash lookup)
2. These matches are removed from MinHash candidate pool
3. MinHash only processes remaining candidates

### Filtering PicHash from Candidates

```python
def filter_pichashes_from_candidate_groups(self, matching_cache, 
                                           candidate_groups, pichash_matches):
    finished_tuples = set()
    for pichash_match in pichash_matches:
        own_fid, foreign_sid, foreign_fid = pichash_match
        if own_fid in candidate_groups:
            # Remove all candidates from the same sample
            if (own_fid, foreign_sid) not in finished_tuples:
                if matching_cache.isSampleId(foreign_sid):
                    pichash_match_sid_fids = matching_cache.getFunctionIdsBySampleId(foreign_sid)
                    candidate_groups[own_fid].difference_update(pichash_match_sid_fids)
                    finished_tuples.add((own_fid, foreign_sid))
    return candidate_groups
```

## Use Cases

### Library Detection

PicHash excels at identifying statically linked libraries:
- Hash all functions from known libraries (e.g., OpenSSL, zlib)
- Query samples against the library index
- Functions with PicHash matches are marked as library code

```python
def getLibraryInfoForSampleId(self, sample_id: int) -> Optional[Dict[str, str]]:
    """Return family_name and version of sample if it is a library
    
    Returns:
        dict with 'family' and 'version' if sample is a library, None otherwise
    """
```

Source: `mcrit/storage/StorageInterface.py:298-308`

### Code Reuse Detection

Identify exact function copies across different malware families:
1. Function appears in multiple samples with same PicHash
2. High confidence (100% match) that code is reused
3. Track code propagation across campaigns

### Small Function Matching

For functions below MinHash threshold (< 10 instructions):
- PicHash provides the only matching mechanism
- Examples: small utility functions, getters/setters, jump tables
- Still achieves high precision for exact matches

</Tip>

## Limitations

**Exact Match Requirement**: PicHash won't match functions with even minor modifications:
- Reordered instructions within blocks
- Different optimization levels
- Manual obfuscation

For these cases, MinHash is more appropriate.

**Collision Possibility**: While rare, hash collisions are theoretically possible. Very different functions could share the same PicHash.

**SMDA Dependency**: PicHash quality depends on SMDA's normalization. Updates to SMDA may change PicHash values, requiring recalculation.
