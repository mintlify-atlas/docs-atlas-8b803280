MCRIT's storage layer manages samples, functions, MinHashes, PicHashes, and their indices. The architecture supports multiple backends and is designed to scale from thousands to millions of functions.

## Overview

The storage layer provides:

- **Data persistence**: Samples, functions, families, and metadata
- **MinHash indexing**: Band-based LSH indices for fast candidate retrieval
- **PicHash indexing**: Hash-to-function mappings for exact matches
- **Query interface**: Unified API across storage backends
- **Caching**: Temporary storage for matching operations

## Storage Backends

MCRIT supports two storage implementations:

### MemoryStorage

**Use Case**: Testing, development, small datasets (< 10,000 functions)

**Characteristics**:
- Pure Python dictionaries and sets
- No persistence (data lost on restart)
- Fast for small datasets
- Simple deployment (no external dependencies)

```python
STORAGE_METHOD = StorageFactory.STORAGE_METHOD_MEMORY
```

### MongoDbStorage (Recommended)

**Use Case**: Production, persistent storage, large datasets (millions of functions)

**Characteristics**:
- Uses MongoDB 5.0+ as backend
- Full persistence and crash recovery
- Indexed queries for scalability
- Supports concurrent access
- Enables distributed deployment

```python
STORAGE_METHOD = StorageFactory.STORAGE_METHOD_MONGODB
STORAGE_SERVER = "127.0.0.1"
STORAGE_PORT = "27017"
STORAGE_MONGODB_DBNAME = "mcrit"
```

## Indexing Strategies

### MinHash Band Indexing

MCRIT uses Locality-Sensitive Hashing (LSH) with banding to index MinHash signatures:

#### Band Projection

A band projection defines which signature positions are used for each band:

```python
def createBandhashProjection(self, minhash):
    band_projection = {}
    banding_strategy = getattr(self._storage_config, "STORAGE_BAND_STRATEGY", "random")
    
    if banding_strategy == "random":
        random.seed(self._storage_config.STORAGE_BAND_SEED)
        band_index = 0
        for band_size, num_bands in self._storage_config.STORAGE_BANDS.items():
            for _ in range(num_bands):
                index_sequence = [index for index in range(len(minhash.getMinHashInt()))]
                random.shuffle(index_sequence)
                band_projection[band_index] = index_sequence[:band_size]
                band_index += 1
    
    elif banding_strategy == "linear":
        # Sequential, non-overlapping bands
        band_size = list(self._storage_config.STORAGE_BANDS.keys())[0]
        num_bands = list(self._storage_config.STORAGE_BANDS.values())[0]
        for band_index in range(num_bands):
            index_sequence = []
            step_size = int(self._minhash_config.MINHASH_SIGNATURE_LENGTH / band_size)
            for index_num in range(band_size):
                index_sequence.append(index_num * step_size + band_index)
            band_projection[band_index] = index_sequence
    
    return band_projection
```

Source: `mcrit/storage/StorageInterface.py:681-717`

#### Band Hash Calculation

For each band, selected signature positions are hashed:

```python
def getBandHashesForMinHash(self, minhash: "MinHash") -> Dict[int, int]:
    if self._band_projection is None:
        self._band_projection = self.createBandhashProjection(minhash)
    
    band_hashes = {}
    minhash_data = minhash.getMinHashInt()
    
    for band_index, permutation in self._band_projection.items():
        # Extract selected positions
        band_data = [minhash_data[i] for i in permutation]
        # Hash to create band hash
        hashed_band_data = MinHash.hashData(band_data, 0)
        band_hashes[band_index] = hashed_band_data
    
    return band_hashes
```

### Linear Banding Strategy (v1.4.6+)

The linear strategy provides better predictability:

- **Band 0**: Positions [0, 16, 32, 48]
- **Band 1**: Positions [1, 17, 33, 49]
- **Band 2**: Positions [2, 18, 34, 50]
- ...
- **Band 15**: Positions [15, 31, 47, 63]

Requirements:
- Single band size definition
- `band_size × num_bands = MINHASH_SIGNATURE_LENGTH`
- Recommended: `{4: 16}` for 64-element signatures

```python
if banding_strategy == "linear":
    if len(self._storage_config.STORAGE_BANDS) != 1:
        raise AttributeError("When using linear strategy, use only a single band_size definition")
    if not band_size * num_bands == self._minhash_config.MINHASH_SIGNATURE_LENGTH:
        raise AttributeError(f"Product of band_size ({band_size}) and num_bands ({num_bands}) "
                           f"must equal MINHASH_SIGNATURE_LENGTH ({MINHASH_SIGNATURE_LENGTH})")
```

Source: `mcrit/storage/StorageInterface.py:75-99`

### Band Matching Process

1. Calculate band hashes for query MinHash
2. For each band, retrieve all functions with matching band hash
3. Count band matches per function
4. Return functions with ≥ `band_matches_required` matches

<Tip>
The default `BAND_MATCHES_REQUIRED=2` provides a good balance. Increase for higher precision (fewer false positives), decrease for higher recall (more potential matches).
</Tip>

## MongoDB Collections

MongoDbStorage organizes data into collections:

- **families**: Family metadata
- **samples**: Sample metadata and statistics
- **functions**: Function entries with MinHashes
- **pichashes**: PicHash to function mappings
- **picblockhashes**: PicBlockHash to function/offset mappings
- **bands_X**: MinHash band indices (one per band ID)
- **xcfgs**: Extended control flow graphs (optional)
- **jobs**: Job queue and history
- **events**: Event log

### Indexing

Critical indices for performance:
```python
# Sample lookups
samples.createIndex({"sha256": 1})
samples.createIndex({"family_id": 1})

# Function lookups
functions.createIndex({"sample_id": 1})
functions.createIndex({"family_id": 1})
functions.createIndex({"pichash": 1})

# Band lookups
bands_X.createIndex({"band_hash": 1})
```

## Performance Considerations

### Indexing Speed
- MinHash generation: ~1-2ms per function
- Band indexing: ~0.1ms per function
- Full database reindex: ~1M functions/hour (parallelized)

### Query Performance
- PicHash lookup: O(1) hash table access
- Band candidate retrieval: O(k) where k = num_bands × avg_functions_per_band
- MinHash scoring: O(n) where n = num_candidates

### Scaling

| Database Size | Recommended RAM | Index Size | Query Time |
|---------------|-----------------|------------|------------|
| 100K functions | 4 GB | ~500 MB | < 1s |
| 1M functions | 16 GB | ~5 GB | < 5s |
| 10M functions | 64 GB | ~50 GB | < 30s |
| 100M functions | 256 GB | ~500 GB | < 5min |

<Tip>
For very large deployments (> 10M functions), consider MongoDB sharding and distributed worker pools.
</Tip>
